## Kode Pertama

```javascript
function processUser(user) {
  if (user) {
    if (user.isActive) {
      if (user.hasPermission) {
        return doSomething(user)
      }
    }
  }
  return null
}
```

---

## Kode Kedua

```javascript
function processUser(user) {
  if (!isValidCandidate(user)) return null;
  return doSomething(user);
}

function isValidCandidate(user) {
  return user && user.isActive && user.hasPermission;
}
```

Apabila harus memilih kode yang akan diperiksa dan diperbaiki pada pukul 01.00 dini hari, saya akan memilih **kode kedua**.

Kode kedua jauh lebih ringkas, alurnya lebih mudah diikuti, dan tidak memerlukan banyak usaha kognitif untuk dipahami. Pada waktu dini hari, kemampuan konsentrasi programmer biasanya menurun, sehingga kode yang lebih sederhana dan tidak berlapis-lapis akan lebih aman untuk dimodifikasi tanpa risiko salah baca atau salah ubah.

## Penjelasan Kode Pertama

Kode pertama mengandalkan pola kondisi bersarang atau **nested if**.

```javascript
function processUser(user) {
  if (user) {
    if (user.isActive) {
      if (user.hasPermission) {
        return doSomething(user)
      }
    }
  }
  return null
}
```

Alur logika pada kode ini berjalan bertahap:

1. Memeriksa apakah objek `user` tersedia.
2. Memeriksa apakah `user.isActive` bernilai `true`.
3. Memeriksa apakah `user.hasPermission` bernilai `true`.
4. Jika ketiga kondisi terpenuhi, fungsi `doSomething(user)` akan dieksekusi.
5. Apabila salah satu kondisi tidak terpenuhi, fungsi mengembalikan nilai `null`.

Secara logika program ini tidak salah, namun dari sudut pandang clean code, struktur bertingkat seperti ini menyulitkan pembacaan karena mengharuskan pembaca memahami setiap lapisan sebelum mengetahui inti dari fungsi tersebut.

---

## Permasalahan pada Kode Pertama

### 1. Indentasi Berlapis yang Berlebihan

Setiap kondisi yang ditambahkan akan mendorong kode lebih ke dalam:

```javascript
if (user) {
  if (user.isActive) {
    if (user.hasPermission) {
      return doSomething(user)
    }
  }
}
```

Pola ini disebut sebagai **arrow anti-pattern**, di mana kode membentuk sudut seperti panah ke kanan. Semakin banyak kondisi, semakin jauh kode bergeser ke kanan dan semakin sulit dibaca.

---

### 2. Tujuan Utama Fungsi Tidak Segera Terlihat

Pembaca harus melewati tiga lapisan kondisi terlebih dahulu sebelum menemukan apa yang sebenarnya dilakukan fungsi ini, yaitu:

```javascript
doSomething(user)
```

Hal ini menambah beban mental bagi siapa pun yang membaca kode, terutama saat dalam kondisi kurang fokus.

---

### 3. Rawan Salah saat Debugging

Ketika terjadi bug di tengah malam, programmer yang kelelahan harus menelusuri setiap lapisan `if` satu per satu. Risiko salah membaca kondisi atau melewatkan kurung kurawal penutup menjadi lebih tinggi.

---

### 4. Sulit Dikembangkan

Jika syarat validasi pengguna bertambah, misalnya:

```javascript
user.emailVerified
user.role === "admin"
user.isNotSuspended
```

Setiap kondisi baru harus ditambahkan sebagai lapisan `if` baru, membuat kode semakin dalam dan semakin rumit.

---

## Penjelasan Kode Kedua

Kode kedua menerapkan pendekatan yang jauh lebih bersih.

```javascript
function processUser(user) {
  if (!isValidCandidate(user)) return null;
  return doSomething(user);
}

function isValidCandidate(user) {
  return user && user.isActive && user.hasPermission;
}
```

Fungsi `processUser()` kini hanya memiliki dua langkah:

1. Jika pengguna tidak lolos validasi, fungsi langsung berhenti dan mengembalikan `null`.
2. Jika pengguna valid, fungsi mengeksekusi `doSomething(user)`.

Teknik ini dikenal sebagai **guard clause**, yaitu pengecekan kondisi tidak valid di awal fungsi agar eksekusi dapat dihentikan lebih awal. Hasilnya, kode inti tidak perlu ditempatkan di dalam blok kondisi yang bersarang.

---

## Mengapa Kode Kedua Lebih Unggul?

### 1. Alur Lebih Langsung dan Mudah Dibaca

```javascript
if (!isValidCandidate(user)) return null;
return doSomething(user);
```

Hanya dengan dua baris, pembaca sudah dapat memahami keseluruhan alur fungsi.

---

### 2. Tidak Ada Nested If

Struktur kode lebih datar sehingga tidak ada lapisan yang membingungkan. Kode yang datar cenderung lebih mudah dipindai secara visual.

---

### 3. Nama Fungsi yang Ekspresif

Validasi dipindahkan ke fungsi terpisah dengan nama yang deskriptif:

```javascript
function isValidCandidate(user) {
  return user && user.isActive && user.hasPermission;
}
```

Nama `isValidCandidate` langsung memberi tahu pembaca bahwa fungsi ini digunakan untuk memeriksa kelayakan pengguna, tanpa perlu membaca isi kondisinya satu per satu.

---

### 4. Lebih Mudah Dirawat

Apabila aturan validasi berubah, cukup ubah fungsi `isValidCandidate()` saja. Fungsi `processUser()` tidak perlu disentuh sama sekali.

Contoh:

```javascript
function isValidCandidate(user) {
  return user &&
    user.isActive &&
    user.hasPermission &&
    user.emailVerified;
}
```

---

### 5. Lebih Aman Dimodifikasi dalam Kondisi Lelah

Pertanyaan tugas sengaja menggunakan konteks "pukul 01.00 dini hari" untuk menekankan pentingnya keterbacaan. Kode yang lebih pendek dan terstruktur jelas akan mengurangi kemungkinan kesalahan saat programmer tidak dalam kondisi optimal.

---

## Prinsip Clean Code pada Kode Kedua

### 1. Guard Clause

```javascript
if (!isValidCandidate(user)) return null;
```

Guard clause menghentikan eksekusi lebih awal sehingga kode inti tidak perlu dikurung dalam blok kondisi bertingkat.

---

### 2. Meaningful Name

Nama `isValidCandidate()` secara eksplisit menggambarkan tujuan fungsi tanpa memerlukan komentar tambahan.

---

### 3. Single Responsibility Principle

Setiap fungsi hanya memiliki satu tanggung jawab:

| Fungsi               | Tanggung Jawab                              |
| -------------------- | ------------------------------------------- |
| `processUser()`      | Menjalankan proses utama terhadap pengguna  |
| `isValidCandidate()` | Memvalidasi apakah pengguna layak diproses  |

---

### 4. Readability

Tanpa indentasi yang berlapis, kode lebih nyaman dibaca bahkan dalam kondisi terburu-buru sekalipun.

---

### 5. Maintainability

Perubahan pada logika validasi cukup dilakukan di satu tempat, sehingga risiko bug akibat perubahan yang tidak konsisten dapat diminimalkan.

---

## Tabel Perbandingan

| Aspek                        | Kode Pertama                               | Kode Kedua                            |
| ---------------------------- | ------------------------------------------ | ------------------------------------- |
| Pola kondisi                 | Nested if bertingkat                       | Guard clause                          |
| Keterbacaan                  | Lebih sulit                                | Lebih mudah                           |
| Kedalaman indentasi          | Dalam (3 tingkat)                          | Datar                                 |
| Proses debugging             | Melelahkan                                 | Lebih efisien                         |
| Pemisahan tanggung jawab     | Bercampur dalam satu fungsi               | Dipisah ke fungsi tersendiri          |
| Kemudahan pengembangan       | Semakin rumit jika kondisi bertambah       | Mudah ditambah tanpa ubah fungsi utama|
| Cocok diperbaiki jam 1 malam | Kurang cocok                               | Lebih cocok                           |

## Kesimpulan

Dari kedua kode yang diberikan, pilihan saya jatuh pada **kode kedua** sebagai kode yang lebih nyaman untuk diperiksa dan diperbaiki pada pukul 01.00 dini hari.

Kode kedua memanfaatkan guard clause untuk keluar lebih awal ketika pengguna tidak valid, sehingga alur utama fungsi tidak terpendam di dalam lapisan kondisi yang bersarang. Selain itu, pemisahan logika validasi ke dalam fungsi `isValidCandidate()` dengan nama yang ekspresif membuat kode jauh lebih mudah dipahami hanya dalam satu kali baca.

Kode pertama tidak salah secara logika, namun struktur nested if-nya menuntut lebih banyak perhatian dan konsentrasi dari pembaca. Dalam situasi kelelahan, struktur seperti itu berisiko menimbulkan kesalahan yang tidak disengaja.

Oleh karena itu, kode kedua lebih mencerminkan prinsip clean code karena lebih sederhana, lebih terstruktur, lebih mudah dikembangkan, dan lebih aman dimodifikasi dalam kondisi apapun.
